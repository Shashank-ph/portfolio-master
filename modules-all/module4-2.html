Development Team Project: Design Document


  • Secure Music Copyright Management Application – Design

    Proposal
    1. Introduction
    This proposal outlines the design for a secure digital platform tailored to manage music copyright artefacts, addressing ongoing challenges in the music industry. The application enables users to securely upload, store, and manage digital content, including lyrics, scores, and audio files. The architecture employs object-oriented design principles, adheres to secure software engineering practices, and complies with standards from ISO/IEC 27000 and the NIST Cybersecurity Framework.

    2. System Overview
    The application serves as a secure repository for copyright owners and artists. It ensures that uploaded artefacts undergo integrity verification via cryptographic hashing, are encrypted with AES-256 encryption for storage, and are timestamped to document ownership and modifications. Users have controlled access to their artefacts, while administrators maintain system-wide authority.

    3. UML Design Diagrams
    Use Case Diagram
    The use case diagram illustrates the interaction between the User and Admin actors and the system. Users can upload, verify, modify, decrypt, and view artefacts. Admins inherit these actions and additionally manage deletion and artefact creation. Key security actions such as checksum verification are embedded within upload routines to automate integrity checks, as encouraged by secure agile development practices (Sharma & Bawa, 2020).

    Class Diagram
    The class diagram demonstrates a modular architecture. The central MusicApp controller manages artefact operations with support from: • ChecksumService for SHA-256 verification,
    • EncryptionService for AES-256 encryption,
    • TimeStampService for audit tracking.
    Classes exhibit strong object-oriented principles, including inheritance (Admin extends User) and composition (MusicApp utilises service classes), which are consistent with the abstraction patterns outlined in Buelta (2022, Chapter 6).

    Sequence Diagram
    The sequence diagram models a typical upload sequence. A user triggers the artefact upload, prompting checksum generation, encryption, timestamping, and database storage. Both Admin and User actors follow this sequence, highlighting shared functionality and reusability, a core tenet of object-oriented design (Cifuentes & Bierman, 2019).

    4. Data and Application Design
    The relational database schema includes:
    • Users: UserID, Name, Role, Email, PasswordHash
    • Artefacts:ArtefactID, UserID, TimestampCreated, TimestampModified Title.
    • Logs: LogID, ActionType, ActorID, Timestamp,EncryptedData, Checksum.
    Artefacts are stored in encrypted format, maintaining ownership through user linkage. Checksum and timestamp fields assure integrity and provenance. The application supports API-based access layered through secure RESTful endpoints and role-based authorisation, in line with best practices in microservices and API security design (Buelta, 2022; Velepucha & Flores, 2023).

    5. Security and Design Patterns
    Security controls include:
    • AES-256 encryption for artefact protection,
    • SHA-256 hashing for integrity verification,
    • Timestamping using cryptographically secure sources,
    • Role-based access to isolate privileges.

    Design patterns implemented:
    • Singleton for database connection handling,
    • Factory pattern for artefact object instantiation,
    • Strategy pattern to encapsulate encryption and checksum logic.
    These architectural approaches align with the secure development patterns recommended in ISO/IEC 27000 (2018) and the NIST Cybersecurity Framework (2020).

    6. Object-Oriented and Agile Principles
    • Encapsulation: Sensitive data is accessed via controlled interfaces.
    • Inheritance: Promotes reusability and logical role hierarchy.
    • Abstraction: Key services are modular and testable.
    • Agility: Iterative development aligns with secure agile models (Sharma & Bawa, 2020), ensuring security is embedded from the outset.

    7. Conclusion
    This design provides a secure and scalable solution to music copyright management. By leveraging object-oriented architecture, UML modelling, and safe design principles, it ensures content protection, legal traceability, and operational transparency. The architecture is future-proofed for modular enhancements and extensibility across roles and content types.

    8. Future Enhancements
    To support long-term scalability and industry adaptation, the application can be extended to include blockchain integration for immutable artefact records and smart contracts for automated licensing. AI-based fingerprinting of audio files could further aid copyright enforcement by detecting duplicates across platforms. Additionally, the platform can evolve to support version control, enabling users to track creative progress and revert changes.
